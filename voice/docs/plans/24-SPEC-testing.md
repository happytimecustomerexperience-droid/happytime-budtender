# 24 — SPEC — TESTING & VERIFICATION — Happy Time Voice Agent

> **Status:** AUTHORITATIVE SPEC (the single test+verification contract for the whole stack). Written 2026-06-22.
> **Reads from (in order, mandatory):** `00-MASTER-ROADMAP.md` → `01-ARCHITECTURE.md` → `02-DECISIONS.md` → `03-CONVENTIONS.md` → the phase doc under test (`10`–`16`) → the sibling specs (`20-SPEC-vapi-deploy.md`, `21-SPEC-budtender-contract.md`, `22-SPEC-kb-seed.md`) → this file.
> **Binds:** every phase doc's "Test plan" section and every spec doc's tests. **This file is the master index of all tests** (Acceptance §13 below requires every phase/spec doc's tests to be enumerated here).
> **Honors ADRs (binding, never contradicted here):** ADR-002 (Squad, not Workflow → no `/workflow` in any test fixture), ADR-003 (idempotent provisioning → the zero-drift gate), ADR-004 (budtender is a separate HTTP service → stubbed/recorded in every voice-repo test), ADR-005 (margin-first/taste-first ranking owned by budtender → the voice repo never re-ranks, asserted), ADR-006 (peppered phone-hash → the PII test), ADR-008 (leak-safe → the **no-leak gate**), ADR-009 (speak OTD → the pricing test), ADR-010 (gpt-4.1-mini assistants / Gemini server-side → the model-shape test), ADR-011 (voice/persona once per member → the no-dup gate), ADR-019 (HMAC fail-closed → the **HMAC-fail-closed gate**), ADR-020 (`voice/tools/` package + registry → tests live per-module).
> **Ports the test chassis from:** `swedish-bot/conftest.py` (the content-aware `FakeGemini` mock), `swedish-bot/pyproject.toml` (`[tool.pytest.ini_options]` markers/addopts), `swedish-bot/tests/test_security.py` + `tests/test_golden.py` (the security + golden-scenario idioms), and `happytime-budtender/budtender/tests/test_no_leak.py` (the allowlist no-leak idiom).
>
> **One-line goal:** every phase ships behind the **same** four-plane gate — Unit · Contract · Provisioning · Manual call — with **two non-negotiable cross-cutting gates** (the **no-leak** contract test and the **HMAC-fail-closed** contract test) re-run on every phase that touches a tool or the webhook, expected values **hand-authored** (never generated by the code under test), and CI = `ruff` + `pytest` + a `provision_vapi --dry-run`. This file enumerates every test in the stack so an autonomous agent can build, run, and verify any phase against a single source of truth.

## 1. Goal & scope

### 1.1 In scope (this spec defines all of)

1. **The six test planes** the whole stack uses — **Unit**, **Webhook-contract**, **Budtender-contract**, **KB-grounding**, **No-leak**, **Security** — mapped onto the four `03-CONVENTIONS.md §5` lanes (Unit / Contract / Provisioning / Manual call) so the `pytest` markers and CI are unambiguous.
2. **The pytest layout + fixtures** — the top-level `tests/` directory (mirroring swedish-bot), the `conftest.py` fixtures (`FakeGemini`, `mock_vapi`, `stub_budtender`, `signed_post`, `seeded_kb`, `voice_call_factory`), the markers (`integration`, `manual`, `live`, `provision`), and `[tool.pytest.ini_options]` config.
3. **Deterministic mock Vapi webhook payloads** — hand-authored JSON fixtures for every event kind the webhook handles: `assistant-request`, `tool-calls` (one per tool: `faq_lookup`, `suggest_products`, `check_inventory`, `pair_upsell`, `notify_vendor_callback`), `status-update`, and `end-of-call-report` (one per outcome). These are the canonical inputs every contract test replays.
4. **GOLDEN voice scenarios** — five scripted transcripts (caller turns → expected tool calls → expected `VoiceCall` outcome), the voice-stack analogue of swedish-bot's `tests/test_golden.py`: a FAQ call, a high-margin suggestion call, a returning-caller personalized call, a vendor no-answer→callback call, and an escalation/defective-return call.
5. **The no-leak assertion** — the exact, reusable `assert_no_leak(blob)` helper + the fixtures it runs against, the structural guarantee the agent cannot speak cost/margin (ADR-008).
6. **CI shape** — `ruff check` + `ruff format --check` + `pytest` (fast lane then contract lane) + a `provision_vapi.py --dry-run` drift check, with the exact gate order and what blocks a merge.
7. **A manual live-call smoke checklist** — the per-phase real-inbound-call definition of done, with what to dial, what to say, and what evidence to paste.
8. **The master test index (§12)** — every named test file from every phase/spec doc, so this file satisfies the acceptance criterion "every phase doc's tests are enumerated here."

### 1.2 Out of scope (owned elsewhere)

- The *implementation* of any feature under test — owned by its phase doc (`10`–`16`).
- The Vapi REST client + provisioner internals — owned by `20-SPEC-vapi-deploy.md` (this file consumes its `mock_vapi` shape and the no-`/workflow` gate).
- The budtender HTTP contract shapes — owned by `21-SPEC-budtender-contract.md` (this file consumes its recorded-response fixtures and the no-leak gate).
- The KB seed content + retrieval — owned by `22-SPEC-kb-seed.md` (this file consumes its `seeded_kb` fixture and the grounding gate).
- Real telephony QA tooling — covered as the **manual** plane only; no automated PSTN harness (a real call is a human step).

### 1.3 Non-negotiable boundaries (binding — a test that violates one is a review block)

- **No network in the fast lane.** `pytest -m "not integration and not manual and not live"` must run offline, deterministic, SQLite-OK. Gemini is mocked (`FakeGemini`), Vapi is mocked (`mock_vapi`), budtender is stubbed (`stub_budtender`). A unit test that opens a socket is a bug.
- **Expected values are hand-authored, never generated by the code under test** (`03-CONVENTIONS.md §5`, `testing.md` discipline). A golden value computed by calling the function it's meant to verify is a tautology and a review block.
- **The two cross-cutting gates are mandatory on every tool/webhook-touching phase:** `assert_no_leak` (ADR-008) and HMAC-fail-closed (ADR-019). They are not optional, not skippable, and re-run per phase against that phase's surfaces.
- **No `/workflow` anywhere** — a test asserts neither `core/services/vapi.py` nor any provisioner references `/workflow` (ADR-002). Fixtures never contain a `/workflow` URL.
- **Per-node config duplication is a test failure** — a test asserts the voice/transcriber/model block + the `DEEPGRAM_KEYTERMS` list appear **exactly once** per assistant payload (ADR-011).
- **The durable `VoiceCall` write is never mocked away in the eocr tests** — the record-survives-failure test asserts the row exists even when the email/summary path raises (ADR-017).
- **Secrets never appear in test output** — a redaction test captures all logs + every result object over a full run and asserts no `VAPI_PRIVATE_KEY`/`VAPI_WEBHOOK_SECRET`/`HHT_BACKEND_TOKEN`/`server.secret` substring.

## 3. The six test planes

The owner asked for six named planes. They map onto the four `03-CONVENTIONS.md §5` execution lanes as follows — the planes are *what we assert*, the lanes are *how pytest runs them*.

| Plane | What it proves | Execution lane (marker) | Network? | Primary source files under test |
|---|---|---|---|---|
| **1. Unit** | Pure functions are correct in isolation — phone-hash, slot validation, OTD formatting, outcome classification, `_clean_graph`, correction reset, cartridge routing, payload builders, tool-arg parsing. | Unit (`-m "not integration and not manual and not live"`) | No (offline, SQLite-OK) | `voice/signing.py`, `voice/recognition.py`, `voice/pricing.py`, `voice/outcomes.py`, `voice/routing.py`, `voice/corrections.py`, `voice/tools/*`, `dashboard/_clean_graph`, `*/provision payload builders`. |
| **2. Webhook-contract** | The Vapi⇄voice boundary: each event kind (`assistant-request`/`tool-calls`/`status-update`/`end-of-call-report`) routes to the right handler and returns the documented JSON shape; unknown tool → `unknown_tool`; idempotent eocr. | Contract (`-m integration`) | Mocked (Vapi client mocked, no real callbacks) | `voice/webhooks.py`, `voice/tools/__init__.py` (registry), `voice/models.py`. |
| **3. Budtender-contract** | The voice⇄budtender boundary: the client builds the right URL/headers/body, known/unknown selects the right call, ≤3 in-stock picks with non-empty `why_this` + `price_otd`, timeout → graceful-empty, the voice repo never re-ranks. | Contract (`-m integration`) | Stubbed/recorded (`stub_budtender`) | `voice/budtender_client.py`, `voice/tools/suggest.py`, `voice/recognition.py`. |
| **4. KB-grounding** | The FAQ surface answers **from KB rows only** (Numbers-Guard): grounded answer == row text; no-match → `grounded:false` + human-offer + **no invented figure**; Mt Vernon → "call to confirm" until O-8. | Contract (`-m integration`) | Mocked Gemini embed (`FakeGemini.embed`) | `kb/semantic.py`, `voice/tools/faq.py`, `kb/vapi_files.py`. |
| **5. No-leak** | cost/margin/velocity/bucket **physically cannot** reach any spoken/serialized/emailed/logged surface (ADR-008). The single most important guarantee. | Contract (`-m integration`), **mandatory gate** | Stubbed (recorded budtender responses that *would* carry cost/margin server-side) | every tool response, every email body, `voice/analytics.py`, `dashboard` render context. |
| **6. Security** | HMAC fail-closed (401 before any handler), prod-fail-closed boot refusal, PII discipline (only the peppered hash persisted), secret redaction, staff-gate on the dashboard, no-`/workflow`. | Mixed: Unit (boot/PII) + Contract (HMAC/redaction/staff-gate), **HMAC gate mandatory** | No / mocked | `voice/signing.py`, `core/middleware.py`, `config/settings.py`, `crm/` PII, `dashboard/` `@staff_member_required`. |

**Marker → lane mapping (the pytest contract):**
- **Fast lane (CI step 1, also pre-commit):** `pytest -m "not integration and not manual and not live"` → Planes 1 + the unit slices of 6 (boot/PII/`_clean_graph`).
- **Contract lane (CI step 2):** `pytest -m "integration and not live"` → Planes 2, 3, 4, 5, and the HMAC/redaction/staff-gate slices of 6. **No real network** — Vapi mocked, budtender stubbed, Gemini mocked.
- **Provisioning lane (CI step 3, dry-run only in CI):** `python tools/provision_vapi.py --dry-run` → the zero-drift + no-`/workflow` + payload-shape checks (a sandbox-key live run is a manual/pre-release step).
- **Manual lane (human, per-phase DoD):** a real inbound call per §11; never in CI.
- **`live` (opt-in, never in CI):** hits real Vertex Gemini (`-m live`) — mirrors swedish-bot's `live` marker; used only to re-validate grounding against the real model before a release.

**The `FakeGemini._classify` retarget (voice roles).** swedish-bot sniffs `"senior Nordland VVS service technician"` etc.; the voice repo sniffs the voice system prompts:
```python
def _classify(system: str) -> str:
    s = system or ""
    if "grounded answer" in s or "answer ONLY from the knowledge base" in s:
        return "faq_grounder"
    if "summarize this call" in s.lower():
        return "call_summary"
    if "editing the SYSTEM PROMPT" in s:        # agent_prompt_assist (14-P4 §3.4)
        return "prompt_assist"
    return "other"
```
with `_default("faq_grounder")` returning a grounded-from-context stub and `_default("call_summary")` a one-line summary — so the multi-call paths work without scripting every call (exactly the swedish-bot property).

### 4.4 `tests/helpers.py` (the shared assertions)

```python
import json, hmac, hashlib
from pathlib import Path

FIXTURES = Path(__file__).parent / "fixtures"
FORBIDDEN = ("cost", "margin", "velocity", "bucket", "margin_pct", "price_z")  # ADR-008 superset

def assert_no_leak(obj) -> None:
    """The ADR-008 gate. Serialize anything and assert no forbidden substring (case-insensitive)."""
    blob = (obj if isinstance(obj, str) else json.dumps(obj, default=str)).lower()
    for word in FORBIDDEN:
        assert word not in blob, f"LEAK: '{word}' reached a customer-facing/serialized surface"

def load_fixture(rel: str) -> dict:
    return json.loads((FIXTURES / rel).read_text(encoding="utf-8"))

def sign(body: bytes, secret: str) -> str:
    """Mirror voice/signing.py's scheme so the test signer == the server verifier."""
    return hmac.new(secret.encode(), body, hashlib.sha256).hexdigest()

def expect_tool_calls(voice_call, names: list[str]) -> None:
    """Assert the ordered set of tool names invoked during a golden run."""
    got = [t.tool_name for t in voice_call.turns.filter(kind="tool").order_by("seq")]
    assert got == names, f"tool-call sequence {got} != expected {names}"
```
> `FORBIDDEN` is a **superset** of budtender's `("margin","cost")` — the voice repo also forbids `velocity`/`bucket`/`price_z` because those are budtender server-only fields (`21-SPEC §5.4`) that must never reach the voice surface either. The budtender source test (`test_no_leak.py`) stays the upstream guarantee; this is the downstream re-assertion.

## 5. Deterministic mock Vapi webhook payloads

Every contract test replays a hand-authored payload from `tests/fixtures/webhooks/`. These are the **canonical inputs**; they are deterministic (no timestamps that matter, fixed ids) so assertions are stable. Shapes follow the Vapi server-message schema the webhook handles (`01-ARCHITECTURE.md §0`, `10-P0 §4`, `20-SPEC §3.4`).

### 5.1 `assistant-request` — Vapi asks which assistant/overrides to use on an inbound call

```json
{
  "message": {
    "type": "assistant-request",
    "call": { "id": "call_test_0001", "phoneNumber": { "number": "+15095550100" },
              "customer": { "number": "+15095550123" } },
    "phoneNumber": { "id": "pn_test", "assistantId": null }
  }
}
```
**Asserted:** the handler returns a 200 with the Squad/assistant selection + hydrated vars (`{{store_name}}`, hours, transfer number) — **never a literal `{{store_name}}`** (export bug #11); the caller number is hashed (peppered) before anything persists.

### 5.2 `tool-calls` — one fixture per tool (the args the assistant passes)

Each carries Vapi's `toolCallId` + `function.name` + `function.arguments`. The handler dispatches by name via `TOOL_REGISTRY` (ADR-020) and returns `{ "results": [{ "toolCallId": "...", "result": <tool output> }] }`.

**`tool_calls_faq_lookup.json`** (P0):
```json
{ "message": { "type": "tool-calls", "call": { "id": "call_test_0001" },
  "toolCallList": [{ "id": "tc_faq_1", "function": {
    "name": "faq_lookup", "arguments": { "query": "what time do you close", "store": "yakima" } } }] } }
```

**`tool_calls_suggest_products.json`** (P1):
```json
{ "message": { "type": "tool-calls", "call": { "id": "call_test_0002",
    "customer": { "number": "+15095550123" } },
  "toolCallList": [{ "id": "tc_sug_1", "function": {
    "name": "suggest_products", "arguments": {
      "store": "yakima", "category": "flower", "effect_desired": "sleep",
      "price_tier": "value", "doh_only": false } } }] } }
```

**`tool_calls_check_inventory.json`** (P1): `arguments: { "store": "yakima", "sku": "SKU-OG-1G" }`.

**`tool_calls_pair_upsell.json`** (P1): `arguments: { "store": "yakima", "anchor_sku": "SKU-FLOWER-35" }`.

**`tool_calls_notify_vendor_callback.json`** (P3):

**Asserted (per tool):** the handler returns the documented result shape; `suggest_products`/`check_inventory`/`pair_upsell` results carry **only** `PUBLIC_PRODUCT_FIELDS` keys (no cost/margin → `assert_no_leak`); `price` is relabeled `price_otd` (OTD, ADR-009); an unknown `function.name` → `{ "error": "unknown_tool" }` (never a 500); the budtender call is built with `phone` iff the caller hash hit a profile (P1 known/unknown switch).

### 5.3 `status-update` — mid-call lifecycle (no durable side-effect)

**Asserted:** the handler 200s, updates the in-flight call-monitor view's status, and writes **no** terminal `VoiceCall` outcome (that is eocr's job). A `status:"forwarding"` update on an escalation/vendor call records the transfer attempt (for `transfer_disposition`).

### 5.5 Fixture invariants (binding)

- **Deterministic ids** (`call_test_000N`), fixed caller numbers in a reserved test range, no wall-clock dependence in assertions.
- **One canonical fixture per (event, tool, outcome)** — tests reference them by name via `load_fixture(...)`; they are never inlined per test (single source of truth, diffable).
- **The leak-bait property:** every budtender fixture (`tests/fixtures/budtender/*.json`) includes a top-level `"cost"` and `"margin"` in a `_server_only` block that the client **must strip** — so `assert_no_leak` failing means the strip regressed, not that the fixture was clean.
- **No `/workflow`** appears in any fixture.

## 6. The no-leak assertion (ADR-008 — the structural guarantee)

The single most important test in the stack. It is the voice-side mirror of budtender's `tests/test_no_leak.py` (which proves `public_product` never serializes cost/margin) — re-asserted at the voice boundary so a regression in *our* mappers (`_speakable_pick`, the tool result builders, email bodies, analytics) is caught even if budtender stays correct.

**The helper** (`tests/helpers.py::assert_no_leak`, §4.4): serialize the object, lowercase, assert none of `("cost","margin","velocity","bucket","margin_pct","price_z")` appears as a substring.

**Where it runs (every surface that could carry a product):**
- `test_leak_guard_p0.py` — `faq_lookup` responses (P0).
- `test_leak_guard.py` — `suggest_products`/`check_inventory`/`pair_upsell` responses (P1, the most important — uses the leak-bait budtender fixture).
- `test_leak_guard_vendor.py` — `notify_vendor_callback` responses (P3).
- `test_sinks_email.py` — every email body (P2).
- `test_no_leak_budtender.py` — every `voice/budtender_client.py` method's raw + tool-shaped output (21-SPEC).
- `test_leak_guard_kb.py` — `faq_lookup` answers + every rendered `vapi_files` markdown file (22-SPEC).
- `test_analytics.py` / `test_leak_guard_p5.py` — `voice/analytics.call_summary` output (P5).
- EXP — every new channel (SMS body, web-chat response, training export) inherits the same assertion (16-EXP §H/§G).

**The proof-of-strip pattern (binding):** the budtender fixture deliberately carries server-only `cost`/`margin`; the test asserts (a) the forbidden substrings are absent from the *output*, AND (b) the *input fixture did contain them* — so the test proves the mapper stripped them rather than the fixture happening to be clean:
```python
def test_suggest_strips_cost_margin(stub_budtender):
    raw = load_fixture("budtender/search_anon.json")
    assert "cost" in json.dumps(raw).lower()          # leak-bait present at source
    out = suggest_products(load_fixture("webhooks/tool_calls_suggest_products.json"))
    assert_no_leak(out)                                # … and gone from the voice output
```

**Mandatory-gate status:** `assert_no_leak` is a **non-negotiable gate** on every phase that touches a tool, an email, or an analytics surface. A phase cannot be "done" without its leak-guard test green (pasted output). It is one of exactly two gates re-run on every such phase (the other is HMAC fail-closed, §8).

## 8. The HMAC-fail-closed gate (ADR-019 — the second mandatory gate)

The other non-negotiable gate. Re-run on **every** phase that touches the webhook or a tool path. It proves a missing/bad Vapi signature is rejected with **401 before any handler runs** — no `VoiceCall` written, no tool dispatched, no email sent — using a **constant-time** compare that **fails closed** when the secret is missing.

**The pattern** (each phase has its own `tests/test_hmac_fail_closed_<phase>.py` but they're identical in shape — re-asserting on that phase's surface):
```python
@pytest.mark.integration
@pytest.mark.django_db
def test_bad_signature_rejected_before_handler(signed_post, client, mock_gemini, stub_budtender):
    payload = load_fixture("webhooks/tool_calls_suggest_products.json")
    bad = signed_post(client, payload, tamper=True)       # flip one byte of the signature
    assert bad.status_code == 401
    assert VoiceCall.objects.count() == 0                  # nothing persisted
    # … and a VALID signature passes through to the handler:
    ok = signed_post(client, payload)
    assert ok.status_code == 200
```
**Where it runs:** `test_hmac_fail_closed_p0.py` (P0, the webhook itself), `test_hmac_fail_closed.py` (P1 tool path), `test_hmac_fail_closed_p2.py` (P2 eocr branch — asserts no `VoiceCall`/no email on a bad sig), `test_hmac_fail_closed_vendor.py` (P3), `test_hmac_fail_closed_p5.py` (P5 cartridge/correction/queue), plus `test_webhook_secret_roundtrip.py` (20-SPEC — the secret the provisioner writes is exactly what the verifier accepts; tampering → 401).

**Boot fail-closed (the unit slice of the security plane):** `test_settings_failclosed.py` — when `DJANGO_DEBUG=0`, the app refuses to boot on a default `DJANGO_SECRET_KEY`, on `PHONE_HASH_PEPPER == DJANGO_SECRET_KEY` (ADR-006), or on a missing `VAPI_PRIVATE_KEY`/`VAPI_WEBHOOK_SECRET`/`HHT_BACKEND_TOKEN`; boots when all are set (`03-CONVENTIONS.md §1.2`).

## 10. CI shape

CI runs on every push/PR; it is the automated definition of "ready to review." The manual call (§11) is the human definition of "ready to ship."

```
┌── CI pipeline (GitHub Actions / equivalent) ───────────────────────────────┐
│ 0. setup:   uv sync                                                         │
│ 1. lint:    ruff check  &&  ruff format --check          ← BLOCKS on fail   │
│ 2. types/migrations (settings-affecting changes):                          │
│             python manage.py check                                          │
│             python manage.py makemigrations --check       ← exit 0 required │
│ 3. fast:    pytest -m "not integration and not manual and not live and not │
│             provision"                                    ← Plane 1 + unit  │
│ 4. contract:pytest -m "integration and not live"          ← Planes 2-6      │
│             (Vapi mocked, budtender stubbed, Gemini mocked — NO real net)   │
│ 5. provision dry-run:                                                       │
│             python tools/provision_vapi.py --dry-run      ← zero-drift +    │
│             (uses a mock/sandbox key; asserts PATCH-only on re-run,         │
│              no /workflow, payload shapes)                 ← BLOCKS on drift │
│ 6. mandatory-gate check (belt-and-suspenders):                             │
│             pytest -m integration -k "leak_guard or hmac_fail_closed        │
│             or no_workflow or secret_redact"              ← must be GREEN    │
└────────────────────────────────────────────────────────────────────────────┘
```

**Gate order + blocking semantics:**
- **Step 1 (ruff) blocks** — no lint-dirty merge (`03-CONVENTIONS.md §1.3`).
- **Step 2 blocks** on `makemigrations --check` ≠ exit 0 (`testing.md` migration gate) for any settings/model change.
- **Steps 3–4 block** on any test failure.
- **Step 5 blocks** on any drift (a re-run that would issue a non-PATCH write) or any `/workflow` reference (ADR-002/003).
- **Step 6** re-runs the two mandatory gates + the security gates explicitly so a green build *cannot* have skipped them (e.g. via a stray `@skip`).

**Pre-commit (local, before push):** `ruff check` + `ruff format --check` + the fast lane (`-m "not integration and not manual and not live and not provision"`) — mirroring swedish-bot's lean pre-commit. The contract lane + provision dry-run run in CI (they need the test DB + the mock surfaces).

**Coverage:** ~90% diff coverage on changed code; ratchet never lowered (`03-CONVENTIONS.md §5`). The new pure modules (`voice/recognition.py`, `voice/outcomes.py`, `voice/routing.py`, `voice/corrections.py`, `voice/pricing.py`, `voice/analytics.py`) carry the highest coverage because they are the deterministic logic the goldens depend on.

**`provision_vapi --dry-run` (the provisioning gate, detail).** The dry-run records-not-issues every Vapi call (ADR-003, `20-SPEC §6`): on an already-provisioned account it must report **0 creates, 0 non-nodrift patches** (zero drift) and print a PATCH-only diff. In CI it runs against the `mock_vapi` surface (no real key needed); a live sandbox-key run is a pre-release manual step. This is the automated proof of idempotency that the manual §11 step 4 re-confirms on the live account.

## 14. Risks / open questions

| Risk / open item | Impact | Mitigation / disposition |
|---|---|---|
| **Golden expected-value drift** — a golden's expected tool-call sequence/outcome is hand-authored; an intentional behavior change makes it red. | False CI failure / churn. | That's the point — a golden is a behavior lock. An intentional change updates `golden_expected.py` **in the same PR** as the behavior, with the diff reviewed. Never auto-regenerate from the orchestrator (tautology, §1.3). |
| **Vapi server-message schema shifts** (the exact `tool-calls`/eocr envelope keys may evolve). | Fixtures + the webhook contract drift from reality. | The fixtures live in one `tests/fixtures/webhooks/` dir; the payload builder + verifier are centralized (`20-SPEC`); a single fixture update + a contract-test re-pin covers a schema bump. The live manual call (§11) is the ground-truth check that catches a real-world divergence CI can't. |
| **`mock_vapi` / `stub_budtender` diverge from the real services** — a contract test passes against a stale stub. | "Green in CI, broken on a call." | The stubs are built from **recorded** real responses (`21-SPEC §5`, the budtender fixtures) and the `20-SPEC` payload shapes; the manual call + the `live` opt-in re-validate against reality before a release; the budtender source `test_no_leak.py` stays the upstream guarantee. |
| **The no-leak `FORBIDDEN` substring list is a blunt instrument** — a legitimate field named e.g. `costume` would false-positive. | A real word trips the gate. | The list is curated to product-economics terms (`cost`/`margin`/`velocity`/`bucket`/`margin_pct`/`price_z`); voice/product vocab has no collision (no "costume" on a dispensary menu). If a collision ever arises, switch that one assertion to a key-set check (keys ⊆ `PUBLIC_PRODUCT_FIELDS`) rather than a substring — already how the shape tests assert. |
| **Duplicate test filenames across phases** (`test_hmac_fail_closed_*`, `test_leak_guard*`, `test_provision_*`, `test_semantic_faq`/`test_vapi_files_mirror` in both P0 and 22-SPEC). | Confusion / a file silently overwriting another. | They are **phase-suffixed and intentionally distinct** (§12 note); P0 ships a first cut of `test_semantic_faq`/`test_vapi_files_mirror`, 22-SPEC owns the canonical KB version — when 22 lands it supersedes P0's stub (one file, the richer assertions). The §10-step-6 `-k` union ensures the gates run regardless of filename. |
| **Coverage ratchet on telephony glue** — `voice/webhooks.py` has hard-to-unit-test Vapi-envelope branches. | Coverage dips below ~90% diff. | Drive the branches via the §5 fixtures in the contract lane (every event kind + every outcome has a fixture), so the envelope handling is covered by replay, not mocked away. The pure logic (`outcomes`/`routing`/`corrections`/`recognition`/`pricing`) is 100%-coverable unit-side. |
| **`live`/`manual` lanes can't run in CI** (real Gemini cost; real PSTN). | A bug only a real call/model reveals slips past CI. | By design: CI is the *automated* gate; the manual call (§11) is the *human* DoD per phase. The `live` marker lets a release run grounding against the real model on demand. Document both as required-before-ship, not required-per-commit. |
| **Provision dry-run needs SOME Vapi surface** even mocked. | The provisioning lane can't run with zero config. | In CI it runs against `mock_vapi` (no real key); the zero-drift + no-`/workflow` + shape assertions are all mock-checkable. The live sandbox-key run is the pre-release step (§11 pre-flight), not a CI step. |
